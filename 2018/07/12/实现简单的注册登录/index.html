实现简单的注册登录

发表于 2018-07-12 | 分类于 HTTP | 阅读次数:

通过实现一个简单的注册登录需求，来引出Cookie在这个过程中所扮演的角色……

★需要做些什么？

◇注册（sign up）页面

  1. 一些表单字段，如一个邮箱、一个密码、一个确认秘密、一个「注册」提交按钮

  2. 添加一些简单的样式

  3. 至于JavaScript的话，则是为整个表单绑定 submit事件，具体逻辑：

    1. 去掉元素的默认动作

    2. 拿到用户输入的数据，用个hash来接收。而且还有为表单做简单的验证，如「某个字段，用户输入数据没有？」「密码是否匹配」等等……

    3. 使用jQuery发送post请求，而hash则作为参数。如果响应回来的是200，则成功，那么就跳转到登录页面；如果响应回来的是4开头的，如400、401、404……那么就告知用户错误信息

      你需要去看一下post的源码才行，如为了知道哪些状态码是回调resolve/reject函数的……

      可看一下这个：Where is jQuery.ajax defined in the jQuery source code?

      注意：服务器返回的数据，那么它是什么格式的，那么前端则需要同后端沟通，制定一个协议，不然二者就要互怼了……暂且称为「解决互怼协议」

      除了规范数据格式以外，这个「解决互怼协议」还有哪些呢？

  4. 在server端，为这个注册页面添加两个路由，一个是GET请求的路由，一个是POST请求的路由；前者很简单，直接响应回一个注册页面即可。后者呢？处理的东西就比较多了，如：

    1. 把从客户端里拿到的数据，解析成原来的hash模样
    2. 加工完数据后，则开始做校验，如「用户填邮箱的时候，是否有@符号啊？」「密码是否配对啊？」「如果前面两点通过了，那么校验邮箱是否已在数据库（db）存在了呢？」 ，如果校验ok的话，那么就把数据写到db里面去，接着就响应个200回去……

◇登录（sign in）页面

  1. 一些表单字段，如一个邮箱、一个密码、一个「登录」提交按钮
  2. 添加点CSS
  3. 还是那些JavaScript-demo，如对数据的校验、发送post请求，不过这次如果登录成功的话则跳转到首页
  4. 在server端，同理，也是需要两个路由，一个是GET请求的路由，一个是POST请求的路由；前者依旧很简单，直接响应回一个登录页面即可。而后者呢？依旧处理的东西比较多，如「加工拿到的数据」、「校验数据」、「查找db里是否存在和邮箱密码匹配的数据」，如果校验成功，找到匹配的数据，那么重点来了，需要添加一个 Set-Cookie的响应头回去。那么我添加了这个头有啥用啊？有以下作用：
    1. 浏览器得到 Cookie 之后，每次请求都要带上 Cookie
    2. 服务器读取 Cookie 就知道登录用户的信息（email）

◇首页（index）

  1. 很简洁的首页，只是为了验证cookie存在的意义，那么如何验证呢？请聚焦server里对此页面的路由
  2. 在server端，只有一个路由——GET请求的路由，毕竟请求该页面的时候，不需要上传什么数据，只是简单的想拿到一个页面而已。这个路由的处理过程是这样子的：
    1. 拿到的Cookie请求头的value值
    2. 加工value值，把拿到的邮箱数据，拿去跟db做比较，如果找到一模一样的，即在db中找到该用户，为了展示cookie的作用，就把首页的占位符替换成用户的密码，然后响应回去。那么你下次想要看首页的时候，就不需要登录后才能看了，直接访问首页即可。不过有个漏洞：由于我们设置的Cookie，是可以通过脚本更改的，那么你在访问首页的时候，更改cookie为其它用户的邮箱（前提是已注册过的），那么就可以访问该用户的首页了

★重新认识下Cookie

◇是什么？

就是一个请求头，告诉server，是哪个天杀的在访问我的首页

◇什么时候设置cookie呢？

server端，登录信息校验成功后，就设置cookie，然后响应回一个带 Set-Cookie的响应头回去，之后你的所有请求都会带有 Cookie

◇demo层面不理解？形象一点？？？

从前有座山，此山很纯朴，需要买票才能进去，而且这里面没有吃的，没有厕所，只有一位扫地僧。也许你会问：「这山不是公开开放的吗？凭啥还要门票？」，扫地僧一扫帚就教你做人……所以你必须买票，不然就是一扫帚……而且这个票可是有期限的哦，如给个两天，每次进山你都得要带票，如果你的票被人捡了，那是你的事儿了，反正扫地僧只认票不认人，你就看着办吧！虽然你脑海中一直回荡着「草泥马」，但只要你表露出这些信息给扫地僧，那么就是一扫帚回应你

这里的「票」就是Cookie了，而「给个两天」则是Cookie存活的时间……

◇Cookie的特点

  1. 服务器通过 Set-Cookie 响应头设置 Cookie
  2. 浏览器得到 Cookie 之后，每次请求都要带上 Cookie
  3. 服务器读取 Cookie 就知道登录用户的信息（email）

◇对Cookie的疑问

  1. 我在 Chrome 登录了得到 Cookie，用 Safari 访问，Safari 会带上 Cookie 吗
    no
  2. Cookie 存在哪
    Windows 存在 C 盘的一个文件里
  3. Cookie会被用户篡改吗？
    可以，下节课会讲 Session 来解决这个问题，防止用户篡改
  4. Cookie 有效期吗？
    默认有效期20分钟左右，不同浏览器策略不同
    后端可以强制设置有效期，具体语法看 MDN
  5. Cookie 遵守同源策略吗？
    也有，不过跟 AJAX 的同源策略稍微有些不同。
    当请求 qq.com 下的资源时，浏览器会默认带上 qq.com 对应的 Cookie，不会带上 baidu.com 对应的 Cookie
    当请求 v.qq.com 下的资源时，浏览器不仅会带上 v.qq.com 的Cookie，还会带上 qq.com 的 Cookie
    另外 Cookie 还可以根据路径做限制，请自行了解，这个功能用得比较少。

★总结

  • 整个登录注册的具体流程，这是需要你去知道的，而具体细节目前不需要深究。总之你跑通这个过程即可

  • 作为一个用户，每次尝试注册登录时候，你得需要想想对面的服务器做了什么，可以提前这么写疑问「为啥我可以登录？而Ta却不能登录呢？」「如果我关闭了浏览器了呢，下次访问该页面还需要重新登录吗？」……

  • 程序员有个特点就是：

    如果我想搞清楚这个事情，那么我会搞清楚这个事情的每一个细节。哪个细节不懂就会去搜，搜到懂为止。

    如果你想搞清楚HTTP下面的TCP、IP什么的，那就去搞，前提是你得先把必须有完成的任务做完，才去深究……

  • 掌握了什么？

    1. 登录怎么做？
    2. 注册怎么做？
    3. 登录之后服务器给浏览器什么东西？——验证数据库，下发Cookie
    4. 注册的时候产生了什么效果？——写数据库
    5. ……还有那问题「用户输入一个地址发生了什么？」万一有Cookie（记录用户ID……）呢？
    6. ……等等

  • 传送门：demo

    下载demo，于目录中运行 node server.js 8888启动服务。为了跑通整个流程，你的访问逻辑是这样的：

    打开注册页面→☞输入信息，点击注册→☞跳转到登录页面→☞填写信息，点击登录→☞跳转到首页

    注意：如果你直接访问首页的话，是无法访问的，而且server也停止了……

  • 这个简单的注册登录，比较简朴，但是会让你很好的明白了，前端和后端都做了哪些事儿，还有一个就是让你知道Cookie到底是个是什么东东？一块小饼干？？？
